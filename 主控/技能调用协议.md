# 技能调用协议

定义系统响应教学资源生成请求时的强制调用规则。工作流顺序以 `主控/工作流注册表.md` 为唯一权威来源。

---

## 1. 调用总则

任何涉及 Skill 输出的任务，必须执行：

```text
读取 AGENTS.md
  ↓
读取 主控/工作流注册表.md
  ↓
读取对应 Skill 定义和检查清单
  ↓
按 主控/输出合约.md 与 主控/技能输出合约.md 生成
  ↓
执行对应 Validator 与 tools/validate_output.py 硬规则检查
  ↓
通过后输出
```

禁止：

- 未读取工作流注册表直接生成。
- 跳过对应 Skill 定义。
- 跳过检查清单。
- 跳过题源字段。
- 自检不通过仍输出。

## 2. `/lesson` 强制链

`/lesson` 的完整步骤见 `主控/工作流注册表.md`。

**前置环节：课题确认**

在执行 `/lesson` 强制链之前，必须先完成课题确认：

```text
解析课时名称（章节号或课时名称）
  ↓
读取 知识库/教材原文/教材原文_教材课时分配.md
  ↓
匹配对应课时信息
  ↓
显示确认信息（章节、节次、课时、教材文件）
  ↓
等待用户确认（用户回复"确认"、"是"、"对"、"正确"等）
  ↓
用户确认 → 进入强制链
用户否认 → 提示重新输入正确的课时名称
```

强制读取：

- `AGENTS.md`
- `主控/工作流注册表.md`
- `主控/输出合约.md`
- `主控/技能输出合约.md`
- `主控/质量门.md`
- `主控/审核协议.md`
- `知识库/教材原文/教材原文_教材课时分配.md`（课题确认阶段）
- `技能/学习目标技能/技能.md` 和 `检查清单.md`
- `技能/评价设计技能/技能.md` 和 `检查清单.md`
- `技能/活动设计技能/技能.md` 和 `检查清单.md`
- `技能/教学设计技能/技能.md` 和 `检查清单.md`
- `技能/提问质量技能/技能.md` 和 `检查清单.md`
- `验证器/学习目标验证器/验证规则.md`
- `验证器/评价验证器/验证规则.md`
- `验证器/活动验证器/验证规则.md`
- `验证器/教学设计验证器/验证规则.md`
- `验证器/教学法验证器/验证规则.md`
- `验证器/数学验证器/验证规则.md`
- `验证器/学情适配验证器/验证规则.md`
- `验证器/一致性验证器/验证规则.md`
- 对应课标、教材、习题、学生近期数据文件

输出：

- `输出/{课时名}_完整教学设计.md`
- `review_status: pending_human_review`

禁止：

- 生成课件。
- 生成课堂提问参考答案。
- 执行课件验证器。
- 自动标记 `human_approved`。
- 在用户未确认课题前直接开始生成教学设计。

## 2a. `/lesson-collab` 强制链

`/lesson-collab` 的完整步骤见 `主控/工作流注册表.md` §2a。

`/lesson-collab` 是 `/lesson` 的对话式协作变体，在知识分析、学习目标、评价设计、活动设计四个关键节点设置确认门，AI 呈现草稿后暂停等待教师审核确认。

**前置环节：课题确认**

同 `/lesson` 的课题确认环节。

**强制步骤链：**

```text
课题确认（同 /lesson）
  ↓
知识分析（AI 生成草稿）
  ↓
🛑 确认门1：知识分析确认
  AI 呈现课型判断、知识点拆解、重难点定位
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
学习目标设计（AI 基于确认的知识分析生成草稿）
  ↓
🛑 确认门2：学习目标确认
  AI 呈现三层学习目标及行为动词
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
评价任务设计（AI 基于确认的目标生成草稿）
  ↓
🛑 确认门3：评价设计确认
  AI 呈现评价任务与目标的对应关系、成功标准
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
问题链设计（AI 自动完成，无需确认）
  ↓
活动设计（AI 基于确认的上游生成草稿）
  ↓
🛑 确认门4-1：教材顺序与模块任务确认
  AI 呈现教材原文内容模块列表，标注每个模块对应的任务类型（读懂/思考/交流/书写/操作等）
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
🛑 确认门4-2：活动步骤逐步确认
  AI 按模块逐个呈现活动步骤，标注每个步骤的时间分配
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
🛑 确认门4-3：台阶提问确认
  AI 呈现完整的问题编号列表（如 ASK_B_01, ASK_M_02, ASK_E_03），每个问题带具体提示和预期回答方向
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
🛑 确认门4-4：整体整合确认
  AI 呈现完整的活动设计整合版（含教材对应位置、题源、升华等）
  → 标注关键决策点
  → 暂停等待教师回复
  → 教师确认后继续 / 教师修改后AI修订再呈现
  ↓
活动质量检查（AI 自动完成）
  ↓
作业设计（AI 自动完成）
  ↓
质量检查（教学设计审核+教学法审核+数学审核+学情适配审核+提问质量审查+一致性校验）
  ↓
输出完整教学设计
  ↓
标记 pending_human_review
```

**确认门交互规范：**

1. AI 必须以结构化格式呈现草稿（使用 `主控/工作流注册表.md` §2a 中的确认门呈现模板）
2. AI 必须明确标注"需要您确认的关键点"（3-5条）
3. 教师确认后才可进入下一步
4. 教师修改意见必须被完整采纳，AI 修订后重新呈现
5. 确认后的内容作为下游的锁定输入，不可回退修改（除非后续环节发现逻辑矛盾）
6. 每个确认门的确认结果记录在教学设计 YAML front matter 的 `collab_gates` 字段

强制读取：

- 同 `/lesson` 强制读取列表
- 额外读取 `主控/工作流注册表.md` §2a 中的确认门交互协议和呈现模板

输出：

- `输出/{课时名}_教学设计.md`
- `review_status: pending_human_review`
- `command: lesson-collab`
- YAML front matter 包含 `collab_gates` 字段

禁止：

- 跳过任何确认门。
- 在教师未确认时继续下游环节。
- 忽略教师修改意见直接使用原草稿继续。
- 将确认门交互过程写入最终教学设计正文。
- 生成课件。
- 生成课堂提问参考答案。
- 执行课件验证器。
- 自动标记 `human_approved`。
- 在用户未确认课题前直接开始生成教学设计。

## 3. `/courseware` 强制链

`/courseware` 的完整步骤见 `主控/工作流注册表.md`。

**前置环节：课题确认**

在执行 `/courseware` 强制链之前，必须先完成课题确认：

```text
解析课时名称（章节号或课时名称）
  ↓
读取 知识库/教材原文/教材原文_教材课时分配.md
  ↓
匹配对应课时信息
  ↓
检查教学设计文件是否存在且状态为 human_approved
  ↓
显示确认信息（章节、节次、课时、教学设计文件、状态）
  ↓
等待用户确认（用户回复"确认"、"是"、"对"、"正确"等）
  ↓
用户确认 → 进入强制链
用户否认 → 提示重新输入正确的课时名称
```

强制读取：

- `AGENTS.md`
- `主控/工作流注册表.md`
- `主控/输出合约.md`
- `主控/图片引用协议.md`
- `主控/审核协议.md`
- `知识库/教材原文/教材原文_教材课时分配.md`（课题确认阶段）
- `技能/分层提问技能/技能.md` 和 `检查清单.md`
- `技能/参考答案生成技能/技能.md` 和 `检查清单.md`
- `技能/课件设计技能/技能.md` 和 `检查清单.md`
- `技能/图片资源技能/技能.md`
- `验证器/课件验证器/验证规则.md`
- `验证器/参考答案验证器/验证规则.md`
- `验证器/图片资源验证器/验证规则.md`
- `输出/{课时名}_完整教学设计.md`
- 对应章节题源文件、学生成绩数据、提问历史记录

前置条件：

- 教学设计必须为 `content_type: lesson`。
- 教学设计必须为 `review_status: human_approved`。

输出：

- `输出/{课时名}_课件.md`
- `输出/{课时名}_课堂提问参考答案.md`

执行流程：

1. 课题确认：解析课时名称，匹配教材课时分配表，检查教学设计状态，等待用户确认
2. 读取并验证人工审核通过的教学设计
3. 调用分层提问技能，分配学生姓名，输出学生选取结果
4. 调用参考答案生成技能，生成参考答案文件
5. 调用课件设计技能，生成课件
6. 依次执行验证：参考答案验证器 → 课件验证器 → 图片资源验证器
7. 验证通过后输出

禁止：

- 重新生成学习目标、评价任务、活动设计或教学设计。
- 消费未人工审核通过的教学设计。
- 使用非同级 `images/` 的图片路径。
- 在用户未确认课题前直接开始生成课件。

## 3a. `/courseware-collab` 强制链

`/courseware-collab` 的完整步骤见 `主控/工作流注册表.md` §4a。

`/courseware-collab` 是 `/courseware` 的对话式协作变体，在**课件结构规划**、**课堂提问设计**、**分层提问分配**三个关键节点设置确认门，AI 呈现草稿后暂停等待教师审核确认。

**前置环节：课题确认**

同 `/courseware` 的课题确认环节。

**强制步骤链：**

```text
课题确认（同 /courseware）
  ↓
读取人工审核通过的教学设计
  ↓
🛑 确认门1：课件结构规划
  AI 呈现课件页面结构框架（含页面内容说明）
  → 教师审核页面顺序和设置
  → 教师确认后继续
  ↓
🛑 确认门2：课堂提问设计
  AI 呈现各页面的问题设计（问题内容、追问设计等）
  → 教师审核提问质量和覆盖度
  → 教师确认后继续
  ↓
🛑 确认门3：分层提问分配
  AI 呈现学生姓名分配方案（基础层/中间层/拓展层对应哪些学生）
  → 教师审核分配合理性
  → 教师确认后继续
  ↓
生成课堂提问参考答案
  ↓
生成 Markdown 课件（基于确认的结构、提问、选人）
  ↓
课件验证
  ↓
图片验证
  ↓
输出课件与课堂提问参考答案
```

**确认门交互规范：**

1. AI 必须以结构化格式呈现草稿（使用 `主控/工作流注册表.md` §4a 中的确认门呈现模板）
2. AI 必须明确标注"需要您确认的关键点"（3-5条）
3. 教师确认后才可进入下一步
4. 教师修改意见必须被完整采纳，AI 修订后重新呈现
5. 确认后的内容作为下游的锁定输入，不可回退修改（除非后续环节发现逻辑矛盾）
6. 每个确认门的确认结果记录在课件 YAML front matter 的 `collab_gates` 字段

强制读取：

- 同 `/courseware` 强制读取列表
- 额外读取 `主控/工作流注册表.md` §4a 中的确认门交互协议和呈现模板

输出：

- 同 `/courseware` 输出
- 额外：YAML front matter 包含 `collab_gates` 字段

禁止：

- 禁止跳过任何确认门。
- 禁止在教师未确认时继续下游环节。
- 禁止忽略教师修改意见直接使用原草稿继续。
- 禁止将确认门交互过程写入最终课件正文。
- 禁止重新生成学习目标、评价任务、活动设计。
- 禁止消费未人工审核的教学设计。
- 禁止使用非同级 `images/` 的图片路径。
- 禁止虚构学生姓名。
- 禁止在用户未确认课题前直接开始生成课件。

## 5. 局部 Skill 任务

局部任务可以单独调用某个 Skill，但仍需：

- 读取 `主控/工作流注册表.md` 确认边界。
- 读取对应 Skill 定义和检查清单。
- 使用基础层、中间层、拓展层。
- 涉及题目时标注 `source_id`、`source_type`、`question_id`。
- 涉及图片时只使用 `./images/文件名`。

---

## 6. 违规处理

| 违规类型 | 处理方式 |
|---------|----------|
| 未读取工作流注册表 | 输出无效，回到规则确认 |
| `/lesson` 生成课件 | 输出无效，删除课件环节 |
| `/courseware` 读取未审核教学设计 | 停止，提示人工审核 |
| 使用禁止词汇 | 回退生成步骤 |
| 缺少三层结构 | 回退生成步骤 |
| 缺少题源字段 | 回退题源探索 |
| 图片路径不合规 | 回退图片准备 |
| 自检不通过仍输出 | 标记为 draft，不得发布 |

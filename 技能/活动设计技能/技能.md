# 活动设计技能

## 技能目标

根据问题链设计促进学习发生的课堂活动。

---

## 输入

* 学习目标（来自 学习目标技能）
* 评价任务（来自 评价设计技能）
* 学情分析（来自 学生近期数据/）
* 问题链设计结果（来自 问题链设计技能）
* 习题分析结果（来自 习题分析技能）
* 对应课时教材参考解答（来自 `知识库/教材参考解答/`，必须存在、课时匹配并通过验证）

---

## 活动设计原则

### 原则1

活动服务问题链。

不是活动服务题目。

---

### 原则2

活动体现：

低起点
高目标
小台阶
快反馈

---

### 原则3

活动必须体现知识形成过程。

避免教师直接讲结论。

---

## 活动类型

- 观察活动：观察数学对象，辨认特征与模式
- 操作活动：动手操作，直观感知数学关系
- 归纳活动：从多个案例中概括规律
- 表达活动：用自己的语言描述数学关系
- 探究活动：在开放情境中探索解决路径
- 论证活动：展示推理过程，论证结论

## 教材顺序约束（强制规则）

教学活动流程必须严格遵循教材原文文本的出示先后顺序，练习和习题环节除外。

### 约束说明

1. 教材正文中的内容模块（如"观察与思考""做一做""大家谈谈"等）具有固定的先后顺序
2. 教学设计中的活动安排不得打乱教材原文内容的先后顺序
3. 打乱顺序指：若教材正文中 A 模块出现在 B 模块之前，则对应 A 模块内容的活动必须在对应 B 模块内容的活动之前
4. 以下教材内容不参与顺序约束：
   - 练习（含"练习"标题标识的课后练习）
   - 习题（含"A组""B组"标题标识的课后习题）
   - 章节引言/章头图等非教学模块内容
5. 不来自教材原文的活动（如认知唤醒、总结迁移、应用拓展）可以灵活插入，但不得使教材原文相关活动的相对顺序发生倒置
6. 每个活动必须在输出模板中标注 `教材对应位置`，明确活动对应教材中的哪个内容模块

### 教材内容模块识别

教材原文中的内容模块通过以下方式识别：
- `##` 级标题（如"## 观察与思考""## 做一做""## 大家谈谈"等）
- 各模块在教材文件中出现的先后顺序即为教材原文顺序

### 违规示例

| 违规情况 | 说明 |
|---------|------|
| 先做"大家谈谈"再讲"做一做" | 教材原文中"做一做"在"大家谈谈"之前 |
| 跳过"观察与思考"直接进入"做一做" | 打乱教材顺序 |
| 习题环节插入正文讲解之间 | 习题不参与排序约束，但不应用习题打乱正文活动顺序 |

## 小台阶推进规则

- 每个活动应有清晰的认知起点，基础层学生也能参与
- 活动步骤之间设置递进台阶，每一步只前进一步，不跳跃
- 每一步台阶对应一组递进式提问，形成"台阶+问题"的链条
- 问题难度随台阶上升而递增，基础层可答首问，中间层可答中间问，拓展层可答末问
- 提问与活动步骤交织推进——不是在活动之前或之后集中提问，而是在步骤转换的关键节点嵌入问题
- **提问对结论负责**：每个台阶的提问必须对后续能否得出本课结论负责——提问的方向、提示的角度、学生的回答，都必须能自然推进到下一步，最终支撑核心结论的得出。如果某个提问无论学生怎么回答都无法引向结论，则该提问无效，必须重新设计
- 台阶提问必须准备具体指向性提示（如"从XX和XX两个角度去看""注意XX的特征"），但课堂呈现必须先让学生独立思考，出现停滞或答偏后再给提示
- 出现理解断裂时应有备用脚手架（拆出更小的子步骤或追加中间问题）

## 问题链与台阶递进

复杂问题的解决过程不是"先展示再提问"或"先提问再展示"，而是台阶与问题**链式交织**：

```
台阶1（最低起点）→ 问题①（基础层可答）→ 台阶2 → 问题②（中间层可答）→ 台阶3 → 问题③（拓展层可答）
```

每条"台阶+问题"形成一个微循环：
1. **台阶**：教师将问题拆解一小步——提示、化条件、引方向，不是给完整解答
2. **问题**：学生在该台阶上思考、回答、推进解决进程
3. **反馈**：即时确认或修正，再进入下一台阶

关键特征：
- **低起点**：第一个台阶从所有学生都能看懂的地方开始
- **高目标**：最后一个台阶指向本课核心结论
- **小台阶**：每步只增加一个认知操作，不超过中间层的承受范围
- **链式交织**：问题是在解决过程中自然生长出来的，不是被"展示"或"提取"的

分层嵌入规则详见 `技能/分层提问技能/技能.md §8`。

---

## 分层规则

| 层级 | 认知要求 |
|------|---------|
| 基础层 | 观察、辨认、模仿 |
| 中间层 | 解释、归纳、表达 |
| 拓展层 | 分析、探究、论证 |

- 同一主题下三层活动应有内在关联
- 基础层活动确保所有人都能参与
- 拓展层活动确保优生有事可做

---

## 快反馈规则

- 每个活动后设置形成性反馈节点
- 反馈方式包括：同伴互评、教师观察、自检清单
- 反馈应针对具体表现，而非笼统评价
- 基础层反馈重点肯定参与
- 中间层反馈重点指出推理完整性
- 拓展层反馈重点推进方法优化

---

## 活动与评价一致性

- 每个活动对应的评价方式应提前设计
- 活动中的关键表现应能被记录作为评价证据
- 活动产出的成果应直接对应学习目标

## 提问编号规范

对于 `/lesson-collab` 流程，所有台阶提问必须统一编号，方便交流确认。

### 编号格式

```
ASK_<层级>_<序号>
```

- 层级：`B` (基础层) / `M` (中间层) / `E` (拓展层)
- 序号：两位数字，从 01 开始递增

### 示例

| 编号 | 层级 | 说明 |
|------|------|------|
| ASK_B_01 | 基础层 | 第 1 个基础层问题 |
| ASK_M_02 | 中间层 | 第 2 个中间层问题 |
| ASK_E_03 | 拓展层 | 第 3 个拓展层问题 |

### 书写规范

每个问题必须包含：
1. **编号**：`ASK_B_01`
2. **提问内容**：完整问题，先呈现给学生
3. **分级备用提示**：不能空泛，仅在学生停滞或答偏时逐级使用
4. **教师参考预期**：说明问题能引导学生得出什么结论，不向学生直接呈现

字段顺序强制为：`提问` → `备用提示` → `教师参考预期`。禁止把提示放在提问之前，以免提前暴露思路。

书写格式：
```
ASK_B_01
  提问: 小红调查的是全班同学，小亮调查的是火车站的人，他们调查的人群有什么不同？
  备用提示1: 比较两处调查的人群范围
  备用提示2: 再看这些人能否代表要研究的总体
  教师参考预期: 能指出调查对象的差异，为后续"样本代表性"作铺垫
```

## 课堂练习/检测规则

### 题目来源规则

课堂练习/检测题目的选题目录来自教材原文：

1. **教材练习栏目**：教材原文中"练习"栏目下的所有题目必须全部选用，不得遗漏
2. **习题A组**：从教材原文"习题A组"中各选取 1 道题目
3. **习题B组**：从教材原文"习题B组"中各选取 1 道题目
4. **总量要求**：课堂练习/检测题目总数量不小于 4 道
5. **补充来源**：当教材练习栏目题目数量不足导致总量不足 4 道时，从 `知识库/练习册题库/` 对应章节补充选题
6. 活动中的单题使用 `source_id`、`source_type`、`question_id`；多道同源题合并共享字段并使用 `question_ids` 列表
7. 选题依据 习题分析技能 的8维度分析报告
8. 分层选用原则：
   - 基础层活动 → 选用标记为"基础层"的习题
   - 中间层活动 → 选用标记为"中间层"的习题，侧重解法比较
   - 拓展层活动 → 选用标记为"拓展层"的习题，侧重变式设计
9. 同一道题可在不同层次活动中以不同方式使用
10. 教材题的预期回答、成功标准、反馈要点和练习答案必须按 `question_id` 与教材参考解答核对；不得自行改写数学结论或答案边界
11. 教材参考解答的 `答案来源` 为 `AI参考推导` 或 `AI参考推导补充` 时，不得表述为教材原文
12. 对应教材参考解答缺失、验证失败或缺少所需 `question_id` 时，立即终止本次工作流

### 环节安排规则

课堂练习/检测环节可根据教学内容灵活安排：

| 安排方式 | 说明 | 适用场景 |
|---------|------|---------|
| 穿插反馈 | 练习题目分散嵌入在教学活动各环节中，讲完一个知识点即时练习、即时反馈 | 新知点较多的课时，每讲完一个模块即用对应练习检测 |
| 集中限时 | 所有练习题目集中在一个时间段内，学生限时独立完成 | 知识点较集中、练习题目关联性强的课时 |

- 两种方式可混合使用：部分练习穿插反馈，部分练习集中限时
- 无论采取哪种方式，每道练习题目都应有明确的反馈节点

### 时间预估规则

- 每道课堂练习/检测题目须标注预估完成时间（格式：`限时 X 分钟`）
- 所有课堂练习题目预估时间之和不得超过课时剩余可用时间
- 时间预估须考虑以下因素：
  - 题目的计算量和书写量
  - 学生分层水平（基础层学生可能需要更多时间）
  - 反馈和订正所需时间
- 时间预估后须保证课时 40 分钟内可完成全部教学环节
- 每个 `ACT-*` 活动必须显式标记 `time_priority: core | flex | backup`
- 每个活动必须使用 `time_budget` 分解 `instruction`、`student_work`、`interaction`、`feedback`、`transition` 和 `total`
- 所有时间使用0.5分钟精度；五个分项之和必须等于 `total`，且与活动标题时间一致
- `core` 是本课目标达成不可删除的任务；`flex` 可根据课堂进度后移；`backup` 仅在时间充足时执行
- 时间验证必须读取 `学生近期数据/教学运行画像.md`；同类活动有3条以上实测记录时，改用 `学生近期数据/课堂活动耗时日志.md` 的实际/计划耗时中位数校准

活动时间字段格式：

```yaml
- time_priority: core
- time_budget:
    instruction: 0.5
    student_work: 3
    interaction: 0.5
    feedback: 0.5
    transition: 0.5
    total: 5
```

### 课下作业规则

- 教材习题A组和B组中未被选为课堂练习/检测的剩余题目，自动进入课下作业池
- 课下作业布置规则详见 `技能/课件设计技能/技能.md §16 课后作业规则`

## 教学材料约束

- 学生可使用的教学材料仅限于：**教材**、**练习册**、**练习本/纸笔**
- 禁止在活动设计中引入额外教学物料：学习单、任务单、检测卡、彩色笔、投影胶片、贴纸等
- 禁止出现"分发""下发""发放""投影展示"等需要额外物料或设备的动作
- 学生的所有书面产出统一在**练习本**上完成，教师反馈通过**板书**或**口头**进行
- 活动步骤中的标注方式应使用**符号**（如箭头、圆圈、下划线）而非不同颜色

---

## 禁止伪探究

- 禁止设置学生无法独立完成的探究任务
- 禁止探究过程缺乏明确指导
- 禁止探究结果与学习目标脱节
- 禁止用简单问答替代数学活动
- 禁止三层活动混在一起无区分
- 禁止教师直接讲结论

## 一题三问（台阶递进式）

### 设计说明

"一题三问"是单题内部台阶递进提问链在活动步骤中的呈现格式。

**设计方法见 `技能/问题链设计技能/技能.md`**，核心原则：

- 每道复杂题的提问链以**思维转折点**为单位设计，不是按"观察→分析→推理"预设模板
- 提问与解题过程交织推进：台阶①（先提问，必要时给备用提示）→ 台阶② → 台阶③
- 基础层在起点台阶参与，中间层推进核心推理，拓展层收束提升

### 书写格式

```
【核心题：台阶递进提问链】
解题路径: 条件 → ?①（思维转折点类型）→ ?② → ?③ → 结论

台阶①:
  台阶: [最小提示]
  提问: "[完整提问]"
  思维转折点: [方向选择/条件定位/缺项补齐/关系建立/方法收束]
  层次: [基础层/中间层/拓展层]

台阶②:
  台阶: [下一步提示]
  提问: "[完整提问]"
  ...

台阶③:
  ...
```

2. [第二步：具体操作]
3. [第三步：具体操作]

- source_id:
- source_type:
- question_id:

同一活动使用多道同源题时，将 `question_id` 改为：

```yaml
- question_ids:
  - [题目编号1]
  - [题目编号2]
```

禁止逗号拼接多个编号，禁止重复输出相同的 `source_id`、`source_type`。

**关联问题链ID填写说明**：填写该活动所服务的问题链ID（如 ChainQ_01），多个问题链用逗号分隔（如 ChainQ_01, ChainQ_02）。

## 自检流程

1. 活动顺序是否严格遵循教材原文文本的先后顺序（练习和习题除外）
2. 每个活动是否标注了 `教材对应位置`
3. 活动是否服务问题链（不是服务题目）
4. 活动起点是否适配学生现有水平
5. 台阶设置是否合理（是否小台阶推进，每个提问是否踩在思维转折点上）
6. 是否有明确的分层设计
7. 每个活动是否有反馈节点
8. 是否避免了伪探究
9. 活动产出是否可评价
10. 核心题是否围绕问题链形成清晰认知台阶（参照 `技能/问题链设计技能/技能.md`）
11. 课堂练习/检测题目来源是否符合规则：教材练习栏目全部题目、习题A组选1题、习题B组选1题，总数 ≥ 4
12. 课堂练习/检测环节安排是否合理（穿插反馈或集中限时），每道题是否有反馈节点
13. 每道课堂练习/检测题目是否标注了预估完成时间
14. 未被选用的教材习题是否已标记为课下作业候选
15. 涉及题目时是否可通过 `question_id` 或 `question_ids` 追踪，多道同源题是否已合并共享题源
